# Fenix Design

This skill covers both creating and implementing designs in pen files. Pen designs are the source of truth for all visual decisions — code follows the designs, not the other way around.

## Design System Ownership

**Pen files are upstream. Code is downstream.**

- The design system lives in pen files (`pens/`)
- `globals.css` and tailwind theme are generated from the design system pen file once at project start
- Only update `globals.css` when the design system itself changes in pen — not on every task
- Screen implementations use the existing CSS variables and theme tokens
- Never derive designs from existing CSS — always design first, then update code to match

## Pencil MCP Tools

### Reading
- `get_editor_state` — check active editor and current selection
- `batch_get` — read nodes by pattern or ID from pen files
- `get_screenshot` — capture visual snapshot for comparison
- `get_guidelines('web-app')` — structural best practices (layout grids, responsive breakpoints, accessibility). Use for structure only, not aesthetics.
- `snapshot_layout` — capture layout structure for verification

### Creating & Updating
- `open_document(filePathOrNew)` — open an existing pen file or create a new one
- `batch_design` — create or update design nodes in pen files
- `find_empty_space_on_canvas` — find space for new frames
- `replace_all_matching_properties` — batch update design tokens
- `set_variables` — set design variables (colors, spacing, typography)
- `get_variables` — read current design variables
- `export_nodes` — export design nodes for reference

### NOT Used
- `get_style_guide` — do NOT use preset style catalogues. The Fenix design system is custom.
- `get_style_guide_tags` — same, not used.

## Design Workflow

### 1. Establish Visual Direction (during `/gsd:discuss-phase`)
- Discuss brand identity, mood, color palette, typography, spacing with the user
- Document decisions in the phase context

### 2. Create Design System (once, at project start)
- Create the design system pen file with:
  - Color palette (primary, secondary, accent, destructive, muted, background, foreground)
  - Typography scale (font families, sizes, weights, line heights)
  - Spacing scale
  - Border radius scale
  - Component patterns (buttons, cards, inputs, navigation, etc.)
  - Dark mode variants
- This is the single source of truth for all visual decisions
- **Created once.** Do not regenerate. Only extend when gaps are found (e.g., a new screen needs a color role or component pattern that doesn't exist yet).

### 3. Design Screens (during planning or as a pre-execution step)
- Create screen designs in pen files matching the requirements
- Use existing design system tokens — do not introduce new colors/fonts/spacing ad hoc
- Follow the page/screen/component architecture:
  - Design the full screen layout (what `_components/screen.tsx` will render)
  - Break out reusable component patterns
- Every required screen should have a pen design before execution starts
- If a screen needs something the design system doesn't cover, extend the design system first (gap fill), then use it

### 4. Implement to Match (during `/gsd:execute-phase`)
- Build components that match the pen designs exactly using the existing theme tokens
- Use `get_screenshot` to compare implementation vs design
- Only update `globals.css` if the design system itself changed in pen — not for regular screen work

### 5. Verify (during `/gsd:verify-work`)
- Use `get_screenshot` and `batch_get` to compare implementation against pen designs
- Design mismatches are BLOCKING
- Check layout, spacing, typography, colors, responsive behavior
- Be specific with file paths and line numbers for issues

## Design Principles

- **Custom design system** — no presets or templates. Design from the project's identity.
- **Dark mode first** for app surfaces
- **Light mode** for public website if appropriate
- **shadcn/ui patterns** — design components that map cleanly to shadcn primitives
- **Responsive** — design for desktop first, ensure mobile breakpoints are considered
- **Accessible** — sufficient contrast, focus states, semantic structure

## Source of Truth (Priority Order)

1. **`.planning/PROJECT.md` + `REQUIREMENTS.md`** — what to build (generated by GSD)
2. **Pen designs** (`pens/`) — how it looks (upstream, drives CSS and components)
3. **`.planning/phases/`** — plans, UI specs, context per phase (generated by GSD)
4. **`DOMAIN_MODEL.md`** — bounded contexts, aggregates, events
5. **`.planning/ROADMAP.md` + `STATE.md`** — phases, progress, decisions

## UI Implementation Rules

- `globals.css` is generated from pen design system at project start — only update when design system changes
- shadcn/ui components — do not build primitives from scratch
- OKLCH color system via CSS custom properties
- Use `cn()` from `@/lib/utils` for conditional class merging
- Geist Sans for UI text, Geist Mono for code/metrics (unless design system specifies otherwise)
