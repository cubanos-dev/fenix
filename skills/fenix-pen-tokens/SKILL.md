# Fenix Pen Tokens

This skill emits CSS custom properties from a pen file's declared variables. It is the canonical bridge between Pencil's design-token layer and the app's stylesheet.

Invoke this skill once per project at bootstrap (after the user has defined pen variables in Pencil), and again whenever pen variables are added or renamed.

## Core rule

**Tokens come from declared pen variables only. Never guess from raw hex.** If a pen has zero variables, stop and tell the user to define them in Pencil first. Guessed tokens produce unstable names and churn every time the pen's colors shift — the whole point of tokens is that the name is stable while the value can change.

## Prerequisites

- A pen file exists under `pens/` (default `pens/app.pen`).
- The user has defined variables in Pencil via the Variables panel. At minimum, a small set of color, spacing, and typography variables should exist.
- `bun run pen:tokens` has been run once — it validates inputs, creates the output directory, and points at this skill.
- Pencil MCP is connected.

## How the user defines variables (one-time, ~5 min)

In Pencil, open the pen file and create variables in the Variables panel:

- Colors: `$bg-section`, `$bg-screen`, `$border-subtle`, `$border-strong`, `$text-primary`, `$text-secondary`, `$text-muted`, `$accent-primary`, `$accent-hover`, `$destructive`
- Spacing: `$space-1` through `$space-12` (or a narrower scale — whatever the design system uses)
- Radius: `$radius-sm`, `$radius-md`, `$radius-lg`
- Typography: `$font-size-sm`, `$font-size-base`, `$font-size-lg`, `$font-size-xl`, `$font-size-2xl`; `$line-height-tight`, `$line-height-normal`, `$line-height-relaxed`
- Font families (optional): `$font-sans`, `$font-display`, `$font-ui`, `$font-mono`

The skill does not care about the exact set — it reads whatever the pen has. But a project is expected to define at least colors and spacing before running this skill; typography can come later.

## Extraction workflow

### 1. Open the document

```
mcp__pencil__open_document(<absolute path to pen file>)
```

Confirm it is active via `mcp__pencil__get_editor_state({ include_schema: false })`.

### 2. Read variables

```
mcp__pencil__get_variables({ filePath: "<pen path>" })
```

The response shape is:

```ts
{
  variables: {
    "bg-section": { type: "color", value: "#E5E1DA" | { value, theme }[] },
    "space-4": { type: "number", value: 16 },
    ...
  }
}
```

If `variables` is empty, **stop** and report: "Pen has no variables. Define them in Pencil first (Variables panel), then re-run."

### 3. Group and emit

Group variables into logical buckets by prefix (`bg-`, `border-`, `text-`, `accent-`, `space-`, `radius-`, `font-size-`, `line-height-`, `font-`). Emit each bucket as a commented section under `:root`.

**For theme-aware variables** (variables whose value is an array of `{ value, theme }`), emit:
- The default (themeless or `light` theme) value under `:root`
- Dark-theme overrides under `:root[data-theme="dark"]` or `@media (prefers-color-scheme: dark)` — match whichever convention the project's existing CSS uses

Example output format (write this to `apps/app/styles/pen-tokens.css` unless `--out` overrode the path):

```css
/* Generated by fenix-pen-tokens skill. Do not edit by hand.
 * Source: pens/app.pen variables. Re-run bun run pen:tokens to refresh.
 */

:root {
  /* Colors */
  --color-bg-section: #E5E1DA;
  --color-bg-screen: #F5F3EF;
  --color-border-subtle: #D1CCC4;
  --color-text-primary: #1B1B1A;
  --color-text-secondary: #5A5754;
  --color-accent-primary: #2E5E4A;

  /* Spacing */
  --space-1: 4px;
  --space-2: 8px;
  --space-4: 16px;
  --space-6: 24px;

  /* Radius */
  --radius-sm: 4px;
  --radius-md: 8px;
  --radius-lg: 12px;

  /* Typography */
  --font-size-base: 16px;
  --font-size-lg: 18px;
  --line-height-normal: 1.5;
}
```

Name mapping:

- Color variables → `--color-<name>` (prefix with `color-` so they don't collide with non-color tokens that share the name)
- Space variables → `--space-<name>` (strip the `space-` prefix from the pen variable if it's already there)
- Radius → `--radius-<name>`
- Font size → `--font-size-<name>`
- Line height → `--line-height-<name>`
- Font family → `--font-<name>`

If a pen variable has a name the skill does not recognize, emit it under a `/* Custom */` section with a `--<name>` property and log it in the output so the user knows to rename if they want it in a typed bucket.

### 4. Wire the output once

After writing the tokens file, the consuming app needs to import it. Check `apps/app/app/globals.css` (or equivalent entry point). If `pen-tokens.css` is not already imported, prepend:

```css
@import './styles/pen-tokens.css';
```

(Or relative path — match the existing import style.)

Do this at most once per project; subsequent skill runs only rewrite `pen-tokens.css`, never the entry point.

### 5. Verify

- The tokens file exists at the expected path.
- Every pen variable from step 2 appears in the output.
- The file compiles (`bun run --cwd apps/app build` or equivalent) — the tokens layer should be pure CSS custom properties with no syntax errors.
- If dark-theme overrides were emitted, the selector matches whatever the app uses (`[data-theme="dark"]`, `@media`, or a class).

## When to re-run

- Any time the user adds, renames, or removes pen variables.
- When values change at scale (the skill re-emits the whole file; individual value changes can also be edited inline by hand, but the skill is idempotent so re-running is always safe).

## Anti-patterns

- **Running the skill against a pen with zero variables and falling back to guessing from raw hex.** Do not. The skill stops; the user defines variables first.
- **Mixing hand-edited tokens with generated tokens.** The file header says "do not edit by hand" for a reason. If a token needs to exist outside the pen variable set, put it in a sibling file and import both.
- **Renaming pen variables without re-running the skill.** The CSS will still reference the old name and break silently. Re-run every time variable names change.
